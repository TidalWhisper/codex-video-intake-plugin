# Creator Trial Product Actions

基于真实样本项目 `creator_trial_20260530_rainy_store` 的再次试用与产物走查，整理当前最值得执行的一轮产品落地清单。

本清单不是工程任务罗列，而是按产品优先级、负责人视角和可验收标准整理，目标是帮助插件从“结构化生产流程”进一步走向“创作者愿意持续使用的创作台”。

## 适用范围

- 试用入口：`README_快速开始.md`
- 推荐主入口：`$video-production-pipeline`
- 回归样本：`video_projects/creator_trial_20260530_rainy_store`
- 重点观察阶段：`Stage 05`、`Stage 06`、`Stage 08`

## 当前核心判断

当前版本已经明显提升了前半段创作拆解和 Stage 05 复核体验，但仍存在一个最高优先级问题：

`系统对外宣称的阶段完成状态` 和 `真实还需要人工处理的状态` 仍然不一致。

这会直接削弱创作者信任，导致用户不敢继续把更多时间押在后半段流程上。

## 总体目标

这一轮不建议继续优先堆新功能，建议先完成四件事：

1. 让状态可信。
2. 让复核顺手。
3. 让等待过程有反馈。
4. 让后半段交付更像真实可推进的成片链路。

---

## P0

### P0-1 单一状态真源

**负责人视角**

当前最伤产品信任的，不是功能缺失，而是多个 manifest 各说各话。用户看到 `project_manifest.json` 像是已经推进到 `STAGE_08_ASSEMBLY_CONFIRMED`，但 Stage 05、06、08 的真实自检又表明还未完全就绪。必须先统一状态真源。

**产品动作**

- 建立一个统一的项目状态裁决器，由阶段证据反推项目状态。
- 禁止 `project_manifest.json` 单独乐观推进 `current_stage`。
- `*_confirmed`、`allowed_next_stage`、`ready_for_*` 必须从真实阶段 evidence 和 self_check 推导。
- 明确区分：
  - `generated`
  - `review_required`
  - `confirmed`
  - `blocked`

**验收标准**

- 当 Stage 05 仍有 blocking review item 时，项目级状态不能显示已可安全进入 Stage 06。
- 当 Stage 06 clip 仍为 draft、pending 或证据不足时，项目级状态不能显示 `STAGE_08_ASSEMBLY_CONFIRMED`。
- 任一项目执行完成后，`project_manifest.json` 与各 stage manifest 的关键状态字段无冲突。
- 对同一个样本重复运行状态汇总，输出结果稳定一致。

### P0-2 阶段闸门硬化

**负责人视角**

现在更像“系统尽量往后跑”，而不是“系统只在准备好时往后跑”。对创作者来说，宁可明确阻断，也不要假装完成。

**产品动作**

- 将 Stage 05 -> 06、06 -> 07、07 -> 08 的推进改成硬闸门。
- 对高风险待复核图片、占位 clip、缺失音频、不可播放 rough cut 一律阻断正式推进。
- 允许生成草稿，但草稿必须带明显状态标签，不能冒充正式通过。

**验收标准**

- Stage 05 存在 `manual_review_cleared = false` 时，Stage 06 只能进入草稿或试跑态，不能进入正式 confirmed 链路。
- Stage 06 `all_required_clips_exist = false` 时，Stage 08 不得写入正式 confirmed 状态。
- UI 或 review 输出里能够明确区分“草稿已生成”和“可正式推进”。

### P0-3 完成状态必须绑定证据类型

**负责人视角**

“完成”这两个字需要变得可信。现在用户很难判断一个 mp4、png、json 到底代表“真实产出”还是“占位痕迹”。

**产品动作**

- 为关键产物统一增加证据标签：
  - `provider_output`
  - `fallback_output`
  - `manual_import`
  - `placeholder_or_incomplete`
- 对 Stage 06、08 的媒体文件补统一 evidence summary。
- 项目主视图显示“当前结果来自哪里”。

**验收标准**

- 每个 Stage 05 图片、Stage 06 clip、Stage 08 粗剪都能明确区分来源类型。
- 占位物不再被统计为正式完成数量。
- 用户不打开 manifest 细节，也能知道当前素材是正式结果还是 fallback 结果。

### P0-4 后半段状态可信回归样本

**负责人视角**

这不是抽象治理任务，必须拿现有最真实的创作者样本回归，而不是重新挑一个更容易的样本。

**产品动作**

- 固定 `creator_trial_20260530_rainy_store` 作为本轮 P0 主回归样本。
- 每次改状态层后，都跑该样本的状态汇总与阶段推进检查。
- 保留一份人类可读的“为什么当前不能继续往后”的解释输出。

**验收标准**

- 样本项目中，项目总状态与 Stage 05/06/08 的真实状态完全对齐。
- 对于不能继续推进的场景，系统能输出明确阻断理由，而不是只留技术字段。

## P1

### P1-1 创作者主视图

**负责人视角**

当前入口仍然太像工程系统。创作者不应该先理解 stage、manifest、runner，才敢开始创作。

**产品动作**

- 建立统一主视图，仅保留五步：
  - 立项
  - 剧本
  - 分镜
  - 关键帧
  - 成片
- 每一步只展示：
  - 当前结果
  - 当前卡点
  - 下一步动作
  - 风险提示
- 默认路径不暴露 manifest 和脚本执行细节。

**验收标准**

- 普通用户不阅读仓库文档，也能知道当前项目卡在哪一步。
- 用户从入口到推进下一步，不需要手动打开多个 stage 目录理解状态。
- 主视图中的状态描述与底层 manifest 一致。

### P1-2 Stage 05 默认工作台化

**负责人视角**

Stage 05 已经有产品雏形，是最值得继续深挖的一步。现阶段最大问题不是“没有建议”，而是仍偏命令式、文档式。

**产品动作**

- 将现有 review queue、repair suggestion、approve flow 收束成默认工作台。
- 画面布局建议：
  - 左侧：当前图片
  - 右侧：风险摘要、检查清单、修复建议
  - 底部：`通过` / `重跑` / `人工替换`
- 保留 Markdown 和脚本作为高级入口，但不作为推荐主路径。

**验收标准**

- 对高风险图片的默认处理不再要求用户先阅读 `manual_review.md` 再敲命令。
- 用户能在同一界面完成看图、判断、通过或重跑。
- 复核后 manifest 状态会同步更新，不需要二次人工维护。

### P1-3 provider 与 fallback 提示前台化

**负责人视角**

用户最怕的不是 fallback，而是不知道现在到底是谁在跑、为什么变慢、是否还值得等。

**产品动作**

- 在主流程里显式提示当前 provider。
- 当 OpenAI 不可用并自动切到 ComfyUI 时，必须给出直白提示。
- 对 fallback 决策轨迹做一行人话总结。

**验收标准**

- 用户在不读日志的前提下，能知道当前是云端跑还是本地跑。
- provider 切换时有明确提示，不再需要从错误日志推断。
- fallback 决策轨迹可在主视图或阶段结果页直接看到。

### P1-4 每阶段统一进度反馈

**负责人视角**

等待焦虑会显著伤害创作体验。系统内部知道很多事，但没有把这些信息翻译给创作者。

**产品动作**

- 每个阶段统一输出三段反馈：
  - 正在做什么
  - 为什么这一步会慢
  - 下一步会产出什么
- 对本地 ComfyUI 阶段增加“可能慢但仍在执行”的可视反馈。

**验收标准**

- 任一耗时阶段开始后，用户都能看到非空的过程反馈。
- 用户能区分“正在执行”和“卡死/失败”。
- 反馈语气偏创作者友好，而不是工程日志口吻。

### P1-5 高风险画面自动二次修复链

**负责人视角**

最耗神的不是“出图”，而是“逐张挑错”。应该优先让系统替创作者多接住一轮。

**产品动作**

- 首批仅覆盖三类高价值错误：
  - 多手/多指
  - 道具数量错误
  - 握持关系错误
- 将风险识别和 prompt 修复建议接成自动二次修复链。
- 对二修结果给出“二修前/后二选一”的快速审图能力。

**验收标准**

- 对 `S001` 这种双人单伞交接镜头，系统能自动发起至少一轮保守修复。
- 二修后结果能够被清晰标记，并与初版并排对比。
- 用户不需要自己先思考怎么改 prompt，系统已提供可执行默认方案。

## P2

### P2-1 Stage 01 文风升级

**负责人视角**

前半段内容已经“正确”，下一步要提升“感染力”。创作者不只需要结构正确，还需要更像成片文案。

**产品动作**

- 为 Stage 01 增加创作语气预设，例如：
  - 治愈日常
  - 情绪反转
  - 导演感独白
  - 高频口播
- 在不丢 story anchors 的前提下，增强语言节奏和短视频语感。

**验收标准**

- 同一 brief 在不同文风预设下能稳定产出风格区分明显的剧本文案。
- 地点、道具、动作、情绪节点不因文风变化而丢失。

### P2-2 Stage 02 导演卡化

**负责人视角**

分镜现在能用，但还不够一眼看懂。创作者希望像看导演卡一样快速判断镜头是否对味。

**产品动作**

- 每个 shot 增加一行“镜头意图摘要”。
- 强化构图重点、情绪作用和叙事作用的可扫读性。

**验收标准**

- 用户扫读 Stage 02 时，不必进入长段文字也能理解每镜头的核心目的。
- `镜头意图摘要` 与 Stage 04 prompt 和 Stage 06 motion route 保持一致。

### P2-3 Stage 04 双层输出

**负责人视角**

目前 prompt 更像给模型看的，不够适合创作者快速确认。应该同时照顾模型和人类。

**产品动作**

- 每个 shot 输出两层内容：
  - 创作者摘要版
  - 模型执行版
- 摘要版强调地点、天气、动作、道具、情绪、构图重点。

**验收标准**

- 创作者可以只看摘要版完成绝大多数 prompt 方向确认。
- 模型执行版继续保留足够的结构化约束和 continuity anchor。

### P2-4 Stage 06-08 默认闭环强化

**负责人视角**

目前已经不像早期那样只有空文件，但还没完全形成“拿到可看粗剪”的成就感。

**产品动作**

- 明确标记哪些镜头来自真实视频、哪些来自关键帧 fallback。
- 强化 rough cut 的“可预览、可解释、可继续推进”能力。
- 为 Stage 08 增加一页人类可读的装配结果摘要。

**验收标准**

- 用户能清楚知道粗剪中哪些段落是正式 clip，哪些段落仍是 fallback reel。
- 粗剪输出时同步附带易读说明，而不只是生成一个 mp4 和 manifest。

### P2-5 快创模式

**负责人视角**

完整流水线适合认真打磨，但不适合高频试题材。需要一条更轻的入门与试创作路径。

**产品动作**

- 提供轻量化模式：
  - 1 段短文案
  - 3 镜头分镜
  - 3 张关键帧
  - 1 版预览
- 允许用户在快创结果基础上升级到完整项目。

**验收标准**

- 用户可在更短时间内完成首轮创意验证。
- 快创模式与完整模式之间可平滑继承已有 anchors 和素材。

---

## 建议执行顺序

1. `P0-1` 单一状态真源
2. `P0-2` 阶段闸门硬化
3. `P0-3` 完成状态绑定证据类型
4. `P1-2` Stage 05 默认工作台化
5. `P1-3` provider 与 fallback 提示前台化
6. `P1-4` 每阶段统一进度反馈
7. `P1-5` 高风险画面自动二次修复链
8. `P1-1` 创作者主视图
9. `P2-1` / `P2-2` / `P2-3` 前半段表达升级
10. `P2-4` / `P2-5` 后半段闭环与快创模式

## 这一轮最关键的成功标志

如果这一轮只看三个结果，最值得盯住的是：

1. 用户不会再看到“项目已到 Stage 08 confirmed，但前面其实还没过审”的矛盾状态。
2. 用户处理 Stage 05 高风险图片时，不需要靠读文档和敲命令完成核心复核。
3. 用户在等待和 fallback 过程中，始终知道系统正在做什么、为什么慢、下一步会给出什么。

## 一句话结论

当前最该做的不是继续横向扩张能力，而是先把“状态可信、复核顺手、等待可感知、粗剪可解释”这四件事打牢。只要这四件事成立，插件就会从“有能力的工程流水线”进一步走向“创作者愿意持续使用的创作产品”。
