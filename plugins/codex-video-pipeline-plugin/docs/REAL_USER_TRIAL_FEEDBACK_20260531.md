# 普通用户试用反馈 2026-05-31

## 试用定位

这次我不是站在开发者视角看仓库，而是站在一个“想赶紧做出一条短视频”的普通创作者视角来用这个 plugin。

我实际新建了一个试做项目：

- `plugins/codex-video-pipeline-plugin/video_projects/video_20260531_222109_project`

试做题材是：

- `下班后的年轻女生在雨夜便利店门口把最后一把伞让给陌生人，离开时发现对方留下一杯热可可`

目标输出选的是：

- `合成粗剪成片`

## 我真实走到哪一步

我实际完成了这些环节：

1. 创建项目目录
2. 创建并锁定 Stage 00 brief
3. 自动生成 Stage 01 剧本
4. 自动生成 Stage 02 分镜
5. 自动生成 Stage 03 人物设定
6. 自动生成 Stage 04 关键帧提示词
7. 生成 Stage 05 关键帧任务与复核说明
8. 尝试继续进入 Stage 06，确认系统是否会误放行

我没有真的做出最终成片。

不是因为系统“假装成功”，而是因为它在 Stage 05 以后开始严格要求：

- 角色参考图
- 真实图片证据
- 人工复核通过

这一点反而是好事，因为它没有骗我说“已经完成”。

## 真实使用感受

## 先说好的

### 1. 前四步非常像一个会整理创意的助理

从一个模糊想法进入剧本、分镜、人物设定、关键帧提示词，这一段是顺的。

感受上不是“空模板”，而是能比较快把一个想法压成可执行结构。

### 2. Stage 05 的风险意识是有产品感的

我很喜欢 Stage 05 里这些内容：

- 高风险镜头优先级
- 风险摘要
- 先看什么
- 第一改法
- 人工复核队列

它不是单纯扔给我一堆 json，而是在努力告诉我“这张图最容易坏在哪里”。

### 3. 后半段闸门是可信的

我在 Stage 05 尚未真正出图、也未过审时，尝试创建 Stage 06 视频片段任务，系统直接阻断：

- `ERROR: keyframe_image_manifest.status must be generated or confirmed before Stage 06`

这点非常重要。

作为普通用户，我宁可被明确拦住，也不希望系统表面继续推进，最后才发现前面全是空的。

## 再说最明显的不舒服

### 1. 用户入口写着“主入口很简单”，但真正用起来还是很工程

前半段还好。

但一到 Stage 05，系统开始大量通过这些方式和我沟通：

- `manifest`
- `review md`
- `python ...py`
- `approve_stage05_review_queue.py`
- `serve_stage05_review_workbench.py`

这时我就明显从“创作者工具”回到了“工程流程工具”。

### 2. 中文项目名最后变成了 `..._project`

我用中文题材名创建项目后，项目目录被命名成：

- `video_20260531_222109_project`

这对普通用户很不友好。

问题不大，但第一印象会掉分，因为项目名失去辨识度，后面找项目会更难。

### 3. Stage 03/04 已经知道我缺角色参考图，但没有直接接住我

系统其实很聪明，已经明确判断出：

- `角色参考图就绪：否`
- `是否可安全自动进入 Stage 05：否`

但它没有把下一步变成一个真正顺手的用户动作，比如：

- `现在为你生成角色参考图入口`
- `现在上传一张主角参考图`
- `跳过参考图，先试跑一版`

它是知道问题的，但还没有把问题包装成一个普通人马上能执行的下一步。

### 4. Stage 05 的 review 很强，但默认路径仍然不是“工作台”

目前的 review 输出已经有明显产品雏形了，但默认仍然是文档 + 命令。

比如它会告诉我去运行：

- `serve_stage05_review_workbench.py`
- `approve_stage05_review_queue.py`

这说明“能力已经有了”，但“默认体验还没切换过去”。

### 5. 错误信息对创作者不够友好

例如：

- `keyframe_image_manifest.status must be generated or confirmed before Stage 06`

这对工程师足够清楚，但对普通创作者来说，最好翻译成人话：

- `你还没有真正拿到并确认关键帧图片，所以现在不能生成视频片段。下一步请先补角色参考图并完成关键帧复核。`

### 6. 系统内部已经有状态判断，但普通用户还看不到一个真正好用的主视图

我看到 `project_manifest.json` 里已经有 `creator_status_overview`，这很对。

但目前它还更像底层状态，不像真正的用户首页。

也就是说，系统已经“知道该怎么对用户说人话”，只是还没有把它做成用户默认看到的东西。

## 我最真实的一句话感受

这个 plugin 现在给我的感觉是：

**前半段已经像创作助理，后半段仍然更像工程生产线。**

它不是不能做，而是“会做”和“好用”之间还差一层产品化包装。

## 我最建议优先改的 6 件事

### P0 先改

1. 做真正的创作者主视图
2. 把 Stage 03 到 Stage 05 中间的“角色参考图补齐”做成显式步骤
3. 把 Stage 05 workbench 变成默认入口，而不是藏在命令里
4. 所有阶段闸门报错都补一层创作者人话解释
5. 把 provider / fallback / manual / placeholder 的来源前台化
6. 优化中文项目标题的自动命名

## 如果只能先改 3 件

如果只允许我站在产品角度选 3 件最值钱的，我会选：

1. `Stage 05 默认工作台化`
2. `角色参考图补齐入口前置`
3. `统一创作者主视图`

因为这三件一旦成立，普通用户会明显感觉：

- 系统知道我在哪
- 系统知道我卡在哪
- 系统知道下一步该怎么接住我

## 结论

我愿意继续用它做短视频，但前提是把“后半段工程感”再压下去一层。

现在它已经不是一个空壳，也不是只会输出文档的假流程。

它已经有真实的内容组织能力、风险意识和闸门意识。

但如果目标是“普通创作者愿意持续使用”，下一步最该补的不是更多能力，而是把已有能力做成默认可感知、默认可点击、默认有人话引导的产品体验。
